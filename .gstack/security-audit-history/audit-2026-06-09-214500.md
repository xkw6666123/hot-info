## Executive Summary

The hot-info project has **2 critical hardcoded secrets** committed to git history and tracked in the repository, posing an immediate risk of API key exposure. The MIMO API key is hardcoded in `_run_asr.py` and the Feishu webhook URL is hardcoded in `feishu_notify.py`. Additionally, the Cloudflare Pages API endpoints use wildcard CORS (`*`), and the trigger-update endpoint lacks authentication, allowing any external party to trigger GitHub Actions workflows. The project has good security headers (CSP, X-Frame-Options, etc.) and properly uses `esc()` for XSS prevention in the frontend, but the secret management issues require immediate remediation.

**Top Priority**: Rotate the exposed MIMO API key and Feishu webhook URL immediately, then remove them from git history.

### [F-003] Unauthenticated Trigger-Update Endpoint (HIGH)

- **Category**: OWASP A01 (Broken Access Control)
- **Severity**: High
- **Confidence**: 9/10
- **Location**: `functions/api/trigger-update.js`
- **Description**: The `/api/trigger-update` endpoint triggers a GitHub Actions workflow dispatch but has no authentication mechanism. Any external party who discovers this endpoint can trigger workflow runs, consuming GitHub Actions minutes and potentially causing denial of service.
- **Exploit Scenario**: 
  1. Attacker discovers the endpoint at `https://pages.github.com/api/trigger-update` (or equivalent Cloudflare Pages URL)
  2. Sends repeated POST requests to trigger GitHub Actions workflows
  3. Exhausts GitHub Actions minutes quota
  4. Could cause data corruption if concurrent runs conflict
- **Reproduction Steps**: 
  1. `curl -X POST https://<pages-url>/api/trigger-update`
  2. Observe GitHub Actions workflow being triggered
- **Remediation**: 
  1. Add a shared secret/token validation to the endpoint
  2. Implement rate limiting (e.g., max 1 trigger per 10 minutes)
  3. Add origin checking or API key validation
  4. Consider adding a CAPTCHA or nonce-based protection
- **Priority**: P1 (this sprint)

### [F-004] Wildcard CORS on API Endpoints (MEDIUM)

- **Category**: OWASP A05 (Security Misconfiguration)
- **Severity**: Medium
- **Confidence**: 9/10
- **Location**: `functions/api/asr.js`, `functions/api/trigger-update.js`
- **Description**: Both Cloudflare Pages API endpoints use `Access-Control-Allow-Origin: '*'`, allowing any website to make cross-origin requests to these endpoints. While the frontend is correctly restricted to `https://rediancha.online` in `vercel.json`, the API endpoints are not similarly restricted.
- **Exploit Scenario**: 
  1. Attacker creates a malicious website
  2. User visits the malicious site while authenticated
  3. Malicious site makes requests to the ASR API or trigger-update API on behalf of the user
  4. Could abuse ASR API quota or trigger unauthorized workflow runs
- **Reproduction Steps**: 
  1. Create an HTML page on any domain
  2. Add JavaScript: `fetch('https://<pages-url>/api/asr', {method: 'POST', body: JSON.stringify({audio: 'test'})})`
  3. Observe the request succeeds with CORS headers
- **Remediation**: 
  1. Replace `'Access-Control-Allow-Origin': '*'` with `'Access-Control-Allow-Origin': 'https://rediancha.online'`
  2. Add Vary: Origin header for proper caching
  3. Consider adding credential requirements for sensitive endpoints
- **Priority**: P1 (this sprint)

### [F-006] GitHub Actions Workflow: Inline Python Scripts (MEDIUM)

- **Category**: OWASP A08 (Software and Data Integrity Failures)
- **Severity**: Medium
- **Confidence**: 7/10
- **Location**: `.github/workflows/auto-update.yml:67-106, 111-150, 155-189`
- **Description**: The GitHub Actions workflow contains large inline Python scripts executed via `python -c "..."`. These scripts are not version-controlled as separate files, making them harder to audit, test, and maintain. They also risk shell injection if variables are interpolated.
- **Exploit Scenario**: 
  1. If any environment variable contains shell metacharacters, the inline Python could break or behave unexpectedly
  2. Difficult to perform code review on inline scripts
  3. No unit test coverage for inline logic
- **Reproduction Steps**: 
  1. Review `.github/workflows/auto-update.yml` lines 67-189
  2. Observe inline Python scripts
- **Remediation**: 
  1. Extract inline Python scripts to separate files (e.g., `scripts/bilibili_asr.py`, `scripts/douyin_asr.py`, `scripts/data_check.py`)
  2. Call them from the workflow: `python scripts/bilibili_asr.py`
  3. This improves audibility, testability, and reduces shell injection risk
- **Priority**: P2 (next sprint)

### [F-007] MD5 Used for ID Generation (LOW)

- **Category**: OWASP A02 (Cryptographic Failures)
- **Severity**: Low
- **Confidence**: 8/10
- **Location**: `generate_hot.py:69`
- **Description**: MD5 is used to generate deterministic IDs: `hashlib.md5(f"{prefix}_{seed}".encode()).hexdigest()[:8]`. While this is not used for security purposes (only for generating unique article IDs), MD5 is a cryptographically broken hash function.
- **Exploit Scenario**: 
  1. Not directly exploitable in this context
  2. If ID uniqueness is relied upon for access control in the future, collisions could be a concern
- **Reproduction Steps**: 
  1. `grep -n "md5" generate_hot.py`
- **Remediation**: 
  1. Consider switching to `hashlib.sha256` for future-proofing
  2. Low priority as current usage is non-security-critical
- **Priority**: P3 (backlog)

### [F-009] GitHub Actions Workflow Permissions (LOW)

- **Category**: OWASP A01 (Broken Access Control)
- **Severity**: Low
- **Confidence**: 7/10
- **Location**: `.github/workflows/auto-update.yml:11`
- **Description**: The workflow has `permissions: contents: write`, which is necessary for pushing commits. However, it does not restrict other permissions, inheriting the default token permissions.
- **Exploit Scenario**: 
  1. If a supply chain attack compromises a dependency (playwright, whisper, opencc), the attacker could leverage the broad default permissions
- **Reproduction Steps**: 
  1. Review workflow file line 11
- **Remediation**: 
  1. Add explicit top-level permissions to restrict the default:
     ```yaml
     permissions:
       contents: write
       actions: none
       packages: none
       ...
     ```
  2. Follow principle of least privilege
- **Priority**: P2 (next sprint)
